# Data Portability Export

Updated: 2026-06-29

This document records the first user-facing data portability slice. The current backend export focuses on ledger data and export metadata, then protects the archive with the user's recently verified secondary PIN.

## Implemented API

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/api/privacy/data-export` | `POST` | Downloads a password-protected user data archive for the authenticated user. |

Request body is optional:

```json
{
  "from": "2026-01-01",
  "to": "2026-12-31"
}
```

When `from` and `to` are omitted, all non-deleted ledger entries are exported.

## Archive Contents

| Path | Description |
| --- | --- |
| `ledger/*.csv` | Ledger CSV generated by the existing ledger export formatter. |
| `metadata/export-metadata.json` | Export timestamp, owner identity, requested date range, included files, and safe counts. |

## Security Rules

| Rule | Reason |
| --- | --- |
| The endpoint uses `@AuthenticationPrincipal` only. | A request body cannot target another user. |
| A recently verified secondary PIN session is required. | Data exports contain sensitive personal financial data. |
| The zip password is the verified secondary PIN. | Matches existing ledger CSV protected export behavior. |
| API keys, signed URLs, prompt payloads, and backup credentials are not included. | Export archives should contain user data, not operational secrets. |
| This first slice excludes binary files/photos. | File archives need storage inventory, size limits, and user-visible progress first. |
