# Household Development History

이 문서는 가계부 영역에서 실제로 겪었던 문제, 원인, 그리고 마지막에 유지하기로 한 해결 방식을 정리한 문서입니다.

중간 시도나 임시 우회보다, 현재 코드 기준으로 남아 있는 구조와 운영 원칙을 중심으로 작성했습니다.

## 1. 초기 문제군

가계부 기능은 단순 CRUD를 넘어서 달력, 빠른 입력, 검색, 휴지통, 통계, 집계 카드, 엑셀/CSV 입출력까지 한 화면에서 이어지는 구조라서 문제가 여러 축으로 동시에 드러났습니다.

주요 문제 축은 아래와 같았습니다.

1. 달력 화면이 화면 크기나 모드에 따라 깨지거나 셀이 잘림
2. 빠른 입력과 수정/취소 흐름이 섞이면서 입력 안정성이 떨어짐
3. 검색/휴지통/긴 목록이 한 번에 길게 렌더되어 조작성이 떨어짐
4. 집계 카드 설정이 사용자 결제수단과 어긋나거나 저장 후 불안정하게 복원됨
5. CSV / Excel 입출력 포맷이 실제 가계부 입력 규칙과 완전히 맞지 않음

## 2. 현재 가계부 구조

현재 가계부는 아래 구조를 중심으로 운영됩니다.

- 프론트 진입 워크스페이스: `HouseholdWorkspace`
- 달력/빠른 입력/집계 UI: `CalendarWorkspace`
- 검색, 관리, 통계 탭: 각 전용 워크스페이스로 분리
- 거래 CRUD / 검색 / 휴지통 / CSV 내보내기: `LedgerEntryService`
- 통계/비교/대시보드 계산: `StatisticsService`
- 사용자 집계 카드 설정 저장: `HouseholdAggregatePreferenceService`
- 월별 Excel 미리보기/가져오기: `LedgerExcelImportService`

즉 현재는 "하나의 거대한 화면에서 모든 계산을 직접 처리"하는 방식이 아니라, `HouseholdWorkspace`가 탭과 anchor date를 잡고, 백엔드는 기능별 서비스로 역할을 나눠 처리하는 구조가 기준입니다.

## 3. 달력 레이아웃과 화면 크기 문제

### 반복 증상

- 달력이 화면 높이에 따라 찌그러짐
- 특정 화면에서 셀이 잘리거나 일부 금액 텍스트가 보이지 않음
- 커스텀 크기 조절과 자동 맞춤 모드가 서로 다른 비율로 동작함

### 원인

- 달력은 한 달 전체 그리드라서 가로/세로 스케일이 동시에 맞아야 하는데, 초기에는 한 축 기준으로만 보정되는 구간이 있었음
- `fit`, `custom`, `preset` 모드가 서로 다른 계산식을 쓰면서 레이아웃 기준이 흔들렸음
- 실제 렌더 높이와 내부 콘텐츠 높이를 별도로 추적하지 않아 셀 잘림이 생김

### 마지막 해결 방식

- 달력 확대/축소는 `CalendarWorkspace` 내부에서 한 기준으로 통일
- `fit`, `custom`, `preset`이 같은 스케일 파이프라인을 타도록 정리
- 자연 크기와 실제 shell 크기를 분리해서 측정
- 세로 레이아웃 보정, 셀 clipping 방지, 최종적으로 균일 scaling 방식으로 통합

### 결과

- 창 크기를 바꿔도 달력 비율이 더 안정적으로 유지됨
- 월 전체/주간 보기/하이라이트 보기에서 셀 손상 가능성이 줄어듦

## 4. 달력 표시 모드와 하이라이트 기준 정리

### 문제

가계부 달력은 단순 날짜 목록이 아니라 "수입/지출/순차액"을 시각적으로 읽어야 해서 표시 기준이 자주 충돌했습니다.

### 마지막 해결 방식

- 하이라이트 모드를 `NET`, `EXPENSE`, `INCOME`으로 명확히 분리
- 기본값은 `NET`
- 주간 보기 모드를 `이번달`, `이번 주`, `이전 주`로 분리
- 집계 week range는 실제 visible month 범위 안에서 clamp

### 결과

- 사용자가 보고 싶은 기준이 명확해짐
- 특정 주간/월간 구간에서 과도한 범위 계산이 줄어듦

## 5. 빠른 입력, 수정, 취소 흐름 안정화

### 반복 증상

- 빠른 입력 중 수정 상태가 남아 다음 입력과 섞임
- 등록 취소와 수정 취소가 같은 흐름처럼 보이며 사용자가 혼란스러움
- 특정 입력 필드가 레이아웃 변경 과정에서 빠졌다가 다시 복구되는 일 발생

### 원인

- 빠른 입력 폼은 즉시 저장과 수정 재진입이 모두 가능해서 상태 분기가 많음
- undo 동작이 "방금 등록"과 "방금 수정"을 동일하게 처리하던 구간이 있었음
- UI 정리 과정에서 `title`, `date` 같은 필수 입력을 공격적으로 줄였다가 실제 사용성 문제가 드러남

### 마지막 해결 방식

- undo 흐름을 등록 취소와 수정 취소로 명확히 분리
- 최근 액션 상태를 별도로 보관해 올바른 undo 라벨과 동작을 보여주도록 보정
- 빠른 입력의 제목, 날짜 입력은 다시 복원
- 소득 입력일 때 결제수단은 `-`를 기본으로 강제해 불필요한 선택 충돌 제거

### 결과

- 빠른 입력과 수정 흐름이 덜 섞임
- 직전 액션 되돌리기 의미가 명확해짐
- 필수 입력 요소가 현재 기준으로 안정적으로 유지됨

## 6. 검색, 휴지통, 긴 목록 처리

### 증상

- 검색 결과와 삭제 항목이 길어질수록 화면이 무거워짐
- 한 번에 너무 많은 결과가 렌더되어 스크롤과 조작성이 떨어짐

### 마지막 해결 방식

- 검색 결과는 페이지 단위 응답으로 변경
- 휴지통 목록도 페이지 단위로 분리
- 비우기, 복구, 영구 삭제를 명확히 나누고 휴지통 액션을 별도 흐름으로 정리

### 결과

- 긴 거래 내역에서도 한 화면에서 다 쏟아지지 않음
- 검색/휴지통 탭의 반응성이 좋아짐

## 7. 사용자 집계 카드 설정 문제

### 반복 증상

- 집계 카드 설정이 저장 후 기대와 다르게 보이거나 비어 보임
- 삭제된 결제수단을 참조한 설정이 남아 UI가 어색해짐
- 카드 종류, 기간, 금액 유형 구성이 일관되지 않음

### 원인

- 사용자별 집계 카드는 JSON으로 저장되는데, 저장된 설정이 현재 활성 결제수단 목록과 어긋날 수 있었음
- 일부 값이 누락되거나 허용되지 않은 값이어도 그대로 복원되면 UI가 흔들림

### 마지막 해결 방식

- 집계 카드 설정은 `HouseholdAggregatePreferenceService`에서 저장/조회 시 모두 normalize
- 허용된 `kind`, `period`, `amountType`만 통과
- 결제수단 기반 카드가 유효하지 않으면 현재 활성 결제수단으로 fallback
- 최대 카드 수는 4개로 고정
- 금액 기준은 `NET`, `INCOME`, `EXPENSE`로 명확히 분리

### 결과

- 잘못 저장된 사용자 설정이 있어도 안정적으로 복원
- 집계 카드가 현재 계정 상태에 맞게 유지됨

## 8. 통계/비교 계산 구조

### 문제

통계 화면은 일별/주별/월별/분기/연간 비교와 카테고리/결제수단 breakdown이 같이 얽히기 때문에, UI에서 그때그때 계산하면 중복 계산과 불일치가 생기기 쉬웠습니다.

### 마지막 해결 방식

- 통계 계산은 `StatisticsService`에서 raw entry를 기준으로 일관 계산
- overview, calendar, category breakdown, payment breakdown, compare, dashboard를 서비스 레벨에서 분리
- 비교 기간 수는 1~24 범위로 제한

### 결과

- 달력, 통계, 대시보드가 같은 기준 데이터로 동작
- 비교 범위가 과도하게 커지는 문제 방지

## 9. CSV 내보내기 문제와 해결

### 증상

- 내보낸 CSV가 다시 가져오기 흐름과 완전히 맞지 않음
- 사용자는 전체 내보내기 외에 "현재 보고 있는 범위"나 "최근 6개월" 같은 선택을 원함
- 민감한 가계부 파일을 평문 그대로 내려받는 것이 부담스러움

### 마지막 해결 방식

- CSV 컬럼 구조를 가져오기와 더 잘 맞도록 정렬
- 내보내기 범위를 `전체`, `최근 6개월`, `최근 1년`, `최근 3년`, `현재 조회 범위`, `직접 선택`으로 지원
- 보안 내보내기는 2차 PIN 확인 후 비밀번호 ZIP으로 생성

### 결과

- 외부 백업/이동용 CSV 활용성이 높아짐
- 사용자가 필요한 범위만 안전하게 내보낼 수 있게 됨

## 10. Excel 가져오기 문제와 해결

### 반복 증상

- 월별 장부 엑셀은 시트 형식이 일정하지 않아 가져오기 실패가 잦음
- 비어 있는 제목/날짜/금액 셀 때문에 preview 결과가 흔들림
- 여러 줄로 감긴 설명이나 시트별 날짜 규칙이 일반 테이블 import와 다름

### 원인

- 실제 사용자 엑셀은 "일반 표"와 "월별 장부 시트"가 섞여 있었음
- 날짜 공란을 이전 행에서 이어받아야 하는 경우가 많았음
- 제목이 비면 그대로 null 처리되어 UX가 불안정했음

### 마지막 해결 방식

- `LedgerExcelImportService`에서 일반 header 기반 import와 월별 장부 시트 import를 모두 지원
- 월 시트는 K:P 열 구조를 따로 읽고, 날짜 공란은 최근 날짜 또는 그 달 1일로 fallback
- 제목 공란은 `작성하지 않음` 기본값 사용
- 중복 가능성이 큰 행은 `날짜 + 제목 + 금액` 기준으로 skip
- 필요한 분류/상세/결제수단은 import 중 자동 생성 가능하게 구성

### 결과

- 사용자가 실제로 쓰던 월별 장부 엑셀도 더 안정적으로 가져올 수 있게 됨
- preview와 import 결과의 일관성이 높아짐

## 11. 모바일/반응형 사용성 보정

### 문제

가계부는 입력 폼, 달력, 집계 카드, 표가 동시에 존재해서 모바일에서 가장 먼저 밀도가 깨졌습니다.

### 마지막 해결 방식

- household 전용 layout과 card/grid 배치를 별도 조정
- 달력 header actions, aggregate controls, sheet layout을 breakpoint별로 재구성
- 제목 overflow, 입력 행 wrapping, 표 영역 간격을 세부 보정

### 결과

- 모바일과 좁은 화면에서 핵심 조작이 덜 겹침
- 입력/조회/집계 영역 분리가 조금 더 명확해짐

## 12. 현재 운영 기준 요약

현재 가계부는 아래 기준으로 유지합니다.

- `HouseholdWorkspace`가 탭과 anchor date를 관리
- 거래 CRUD / 검색 / 휴지통 / CSV export는 `LedgerEntryService`
- 통계와 비교 계산은 `StatisticsService`
- 집계 카드 설정은 `HouseholdAggregatePreferenceService`가 normalize 후 저장
- Excel import는 preview 후 선택 import를 기본으로 사용
- 달력은 `fit`, `custom`, `preset`을 지원하지만 같은 scaling 기준으로 유지

## 13. 검증 방식

가계부 관련 변경은 가능한 한 아래 기준으로 검증했습니다.

- 프론트: `cmd /c npm run build`
- 백엔드: `.\gradlew.bat test`
- 수동 검증
  - 빠른 입력 등록/수정/취소
  - 달력 화면 크기 변경과 fit/custom preset 전환
  - 통계 범위 이동과 비교 단위 변경
  - 집계 카드 저장 후 새로고침 복원
  - CSV 내보내기와 Excel preview/import 흐름
