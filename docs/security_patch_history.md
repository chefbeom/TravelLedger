# Security Patch History

이 문서는 현재 저장소에서 실제로 반영된 보안 패치 작업을 정리한 문서입니다.

과장된 "보안 강화" 목록이 아니라, 운영 중 드러난 위험을 어떤 방식으로 줄였는지와 지금 유지 중인 기준을 중심으로 기록합니다.

## 1. 초기 위험 축

운영 중 특히 민감했던 축은 아래와 같았습니다.

1. 인증이 단순 비밀번호만으로 끝나 관리자/중요 작업 보호가 약함
2. 로그인 실패를 무제한으로 허용하면 추측 시도에 취약함
3. CSV / Excel / 업로드 파일이 외부 입력이라 수식 주입, 과도한 텍스트, 위험한 파일명 같은 문제가 생길 수 있음
4. 백업/복구는 관리자 기능이지만 잘못 다루면 서비스 전체에 영향이 커서 별도 보호가 필요함
5. 브라우저와 프록시 사이에서 CSRF, 세션, 프레임 정책, 공개 에러 메시지 노출을 조심해야 함

## 2. 인증 구조 강화

### 최종 기준

- 로그인은 `비밀번호 + 2차 비밀번호(숫자 8자리)`를 함께 검증
- 비밀번호와 2차 비밀번호는 모두 해시 저장
- 프로필 비밀번호 변경과 2차 비밀번호 변경도 현재 2차 비밀번호 검증 후에만 허용

### 구현 기준

- `AppUserService`에서 2차 비밀번호를 숫자 8자리로 정규화
- 로그인 성공 후에는 검증된 2차 비밀번호를 세션에 저장
- 민감한 작업은 별도 2차 검증 또는 관리자 확인 흐름을 사용

### 결과

- 비밀번호가 노출되더라도 2차 방어선이 하나 더 생김
- 계정 설정 변경에도 같은 보호 기준을 적용

## 3. 로그인 실패 제한과 감사 로그

### 문제

로그인 실패를 계속 허용하면 무차별 대입 시도에 취약하고, 운영자는 어떤 실패가 발생했는지 추적하기 어렵습니다.

### 최종 해결 방식

- 같은 IP에서 로그인 실패 5회 이상이면 24시간 차단
- 차단 상태는 메모리뿐 아니라 Redis state에도 저장 가능하게 구성
- 로그인 성공 / 비밀번호 실패 / 2차 비밀번호 실패 / 차단 상태를 감사 로그로 기록
- 사용자에게는 세부 원인을 노출하지 않고 "로그인 정보가 올바르지 않습니다" 수준으로 응답

### 결과

- 추측 시도의 반복 비용이 커짐
- 운영자는 실패 패턴을 로그로 볼 수 있지만, 외부에는 과도한 정보를 내보내지 않음

## 4. 관리자 접근 보호

### 문제

일반 로그인만으로 곧바로 관리자 기능까지 열리면, 관리자 영역 보호 수준이 충분하지 않습니다.

### 최종 해결 방식

- 관리자 API는 기본적으로 `ADMIN` 권한이 필요
- 그 위에 관리자 페이지 접근은 별도의 admin access verify 흐름으로 한 번 더 검증
- 초대 링크 발급도 관리자만 가능하도록 제한
- 운영 화면에서는 불필요한 관리자 힌트 문구와 세부 실패 정보를 줄임

### 결과

- 관리자 기능은 "로그인됨"과 "관리자 접근 검증 완료"를 분리한 구조가 됨
- 관리자 전용 기능 노출면이 줄어듦

## 5. Spring Security / 브라우저 보호 기준

### 현재 적용 기준

- CSRF 토큰은 쿠키 기반으로 발급하고 프론트가 헤더로 전달
- 인증이 필요한 API는 기본적으로 세션 인증이 필요
- 인증 실패는 JSON 401, 권한 부족은 JSON 403으로 정리
- `X-Frame-Options: SAMEORIGIN`
- remember-me는 서버 저장형 persistent token 기반 사용
- request cache는 비활성화해 불필요한 redirect 성격 저장을 줄임
- forward header 전략은 프록시 환경 기준으로 framework 처리

### 결과

- 프론트 fetch 기반 SPA와 맞는 CSRF / 세션 구조 정착
- 보안 헤더와 일관된 오류 응답 기준 확보

## 6. CSV 내보내기 보안 패치

### 문제

CSV는 엑셀에서 바로 열리는 경우가 많아서, 값이 `=`, `+`, `-`, `@`로 시작하면 formula injection 문제가 생길 수 있습니다.

### 최종 해결 방식

- CSV formatter에서 셀 시작 문자가 `=`, `+`, `-`, `@`이면 앞에 `'`를 붙여 수식 평가를 막음
- 줄바꿈과 큰따옴표도 escape 처리
- 민감한 가계부 CSV는 2차 PIN 확인 후 password-protected ZIP으로 내려받는 보호 경로도 제공

### 결과

- CSV export를 열 때 수식 실행 위험을 줄임
- 내보내기 파일 자체도 평문보다 안전한 전달 방식 선택 가능

## 7. 텍스트 입력 sanitize

### 문제

거래 제목/메모나 import된 텍스트가 지나치게 길거나 예기치 않은 형태면 저장/표시 안정성이 떨어집니다.

### 최종 해결 방식

- `LedgerEntryTextSanitizer`에서 제목과 메모 길이 제한
- 제목 overflow는 잘라내고, 넘친 전체 제목은 memo로 이동
- Excel import 메모에서 import 전용 안내 문구는 별도로 strip 가능하게 정리

### 결과

- 저장 필드 길이가 통제되고 UI overflow 가능성이 줄어듦
- import 텍스트 정리가 일관됨

## 8. 파일 업로드와 파일명 sanitize

### 문제

파일 업로드는 경로 조작, 이상한 파일명, 원본 바이너리 집중 업로드 같은 위험을 함께 가집니다.

### 최종 해결 방식

- 여행 업로드는 프론트 준비 + presigned URL + 스토리지 직접 업로드 구조로 이동
- 백엔드는 presigned URL 발급과 메타데이터 저장 중심으로 축소
- travel / family album 저장소 서비스에서 파일명은 영문/숫자/`._-`만 남기는 safe file name으로 normalize
- presigned 업로드는 명시적으로 켜진 경우에만 사용

### 결과

- 백엔드가 원본 업로드 본문을 직접 오래 들고 있지 않음
- 파일명 기반 위험을 줄이고 스토리지 키 형식을 통일

## 9. 백업/복구 보호

### 문제

백업/복구는 관리자 기능 중에서도 가장 파급 범위가 커서, 동시 실행이나 잘못된 restore가 매우 위험합니다.

### 최종 해결 방식

- 백업/복구 작업은 exclusive operation lock으로 단일 실행
- restore 중에는 `RestoreMaintenanceFilter`가 일반 API를 `503`으로 막고, health/admin access 등 꼭 필요한 경로만 통과
- 백업 파일명 검증과 작업 디렉터리 분리
- MinIO backup archive는 zip entry 경로를 normalize하고 `..` 시작 경로를 차단
- Google Drive quota fallback 등 외부 실패도 예외 메시지를 정리해서 처리

### 결과

- 복구 중 일반 사용자 요청이 섞여 데이터 정합성을 더 해치지 않도록 방지
- backup archive 생성 시 path traversal 위험 축소

## 10. 관리자 화면 노출면 축소

### 문제

관리자용 로그인 로그, 문의, 백업 목록은 편리하지만, 너무 많은 세부 정보를 한 번에 보여주면 운영 면에서 불필요한 노출과 과부하가 생깁니다.

### 최종 해결 방식

- 관리자 로그인 로그 / 문의 / 목록 화면은 페이지 단위로 정리
- 인증 실패 상세를 과도하게 노출하지 않도록 조정
- 답변 완료 문의는 archive 성격으로 분리

### 결과

- 관리자 화면에서 필요한 정보만 단계적으로 확인 가능
- 민감한 실패 이유 노출을 줄임

## 11. 배포/인프라 보안 기준

### 현재 문서화된 기준

- HTTPS reverse proxy 기준 운영
- presigned URL은 외부에서 접근 가능한 HTTPS 주소만 사용
- MinIO 공개 주소와 root credential 불일치 시 presigned 업로드를 막음
- DB / MinIO / Redis를 역할별로 분리 배포 가능하게 유지
- project tenant provisioning 시 DB 권한은 전체가 아닌 해당 프로젝트 DB로 한정

### 결과

- 운영 환경에서 공개 주소, 자격 증명, 권한 범위가 문서와 코드 기준으로 맞춰짐

## 12. 현재 운영 보안 기준 요약

현재 코드 기준 보안 운영 원칙은 아래와 같습니다.

- 로그인은 비밀번호 + 2차 비밀번호
- 로그인 실패는 IP 기준 제한
- 관리자 페이지는 별도 검증 흐름 사용
- CSRF, 세션, remember-me, frame option을 명시적으로 설정
- CSV export는 formula injection을 sanitize
- 텍스트 입력은 길이와 형식을 sanitize
- 업로드 파일명은 safe file name으로 normalize
- 백업/복구는 단일 실행과 maintenance gate를 사용

## 13. 검증 방식

보안 패치 성격의 변경은 가능한 한 아래 기준으로 검증했습니다.

- 백엔드 테스트: `.\gradlew.bat test`
- 프론트 빌드: `cmd /c npm run build`
- 수동 확인
  - 로그인 성공 / 실패 / 차단 흐름
  - 2차 비밀번호 검증 및 변경
  - 관리자 접근 검증
  - CSV export 파일 열기
  - restore 중 일반 API 차단
  - presigned 업로드 성공/실패 구성 확인

## 14. 문서 목적
