# Security Patch History

이 문서는 현재 저장소에서 실제로 반영된 보안 패치 작업을 공개 가능한 수준으로 정리한 문서입니다.

실제 운영 서버 보호를 위해 임계값, 내부 탐지 규칙, 일부 운영 절차, 세부 경로와 같은 민감한 정보는 의도적으로 문서에서 제외하거나 추상화했습니다.

## 1. 정리 목적

보안 문서는 "무엇을 어떻게 공격할 수 있는지"를 설명하는 문서가 아니라, 운영 중 어떤 위험 축을 줄였고 지금 어떤 원칙으로 서비스를 유지하는지를 기록하는 문서여야 합니다.

이 문서는 아래 두 목적에 맞춰 작성합니다.

1. 팀 내부에서 보안 관련 개발 흐름을 회고할 수 있을 것
2. 공개 저장소와 위키에 올려도 운영 중인 서버의 취약점 정보가 직접 노출되지 않을 것

## 2. 주요 보안 작업 축

현재까지의 보안 패치 작업은 크게 아래 축으로 정리할 수 있습니다.

1. 인증과 계정 보호 강화
2. 관리자 기능 접근 보호
3. 브라우저/세션/요청 보호 기준 정리
4. 파일/텍스트/내보내기 입력값 보호
5. 백업/복구 같은 고위험 운영 작업 보호
6. 공개 화면과 운영 화면에서의 민감 정보 노출 축소

## 3. 인증과 계정 보호

### 문제

생활 기록 서비스는 가계부, 여행 기록, 가족 사진처럼 민감한 데이터를 함께 다루기 때문에, 단일 인증 수단에만 의존하면 계정 보호 강도가 부족합니다.

### 최종 유지 기준

- 로그인은 다단계 검증 구조를 사용합니다.
- 민감한 계정 변경 작업은 추가 검증을 거치도록 설계합니다.
- 인증 정보는 원문 저장이 아니라 안전한 해시 기반 저장을 원칙으로 합니다.

### 정리

이 영역의 핵심은 "로그인 성공 여부"만 보지 않고, 계정 설정 변경과 민감한 데이터 반출에도 같은 수준의 보호를 이어서 적용하는 것입니다.

## 4. 로그인 시도 제한과 감사 추적

### 문제

로그인 실패를 무제한 허용하면 추측 시도와 반복 시도에 취약해질 수 있고, 운영자는 어떤 실패 패턴이 있었는지 추적하기 어렵습니다.

### 최종 유지 기준

- 반복 실패 시도에 대한 차단 또는 지연 장치를 둡니다.
- 로그인 관련 이벤트는 감사 추적이 가능하도록 기록합니다.
- 외부 응답에는 과도한 실패 상세를 노출하지 않습니다.

### 정리

즉 운영자는 내부적으로 실패 유형을 구분할 수 있어야 하지만, 외부 사용자는 계정 유무나 실패 세부 원인을 쉽게 유추할 수 없도록 유지하는 것이 기준입니다.

## 5. 관리자 기능 보호

### 문제

관리자 기능은 일반 사용자 기능보다 훨씬 높은 파급력을 가지므로, 일반 로그인 상태와 동일한 수준으로 다루면 안 됩니다.

### 최종 유지 기준

- 관리자 기능은 권한 기반 접근 제어를 전제로 합니다.
- 고위험 관리자 화면과 작업은 추가 검증 단계를 둡니다.
- 관리자 전용 기능과 운영 도구는 공개 영역에서 불필요하게 드러나지 않도록 정리합니다.

### 정리

관리자 보호는 "관리자 역할이 있다"만으로 끝내지 않고, 실제 운영 도구 접근에서도 한 번 더 안전장치를 두는 구조를 기준으로 삼습니다.

## 6. 브라우저 / 세션 / 요청 보호

### 문제

SPA 환경에서는 브라우저와 API 사이에서 세션, 쿠키, CSRF, 프록시 헤더, 보안 헤더가 함께 맞물립니다. 이 기준이 흐트러지면 인증 자체가 불안정해지거나 요청 위변조 방어가 약해질 수 있습니다.

### 최종 유지 기준

- 상태 변경 요청에는 CSRF 보호를 적용합니다.
- 세션 기반 인증과 브라우저 보안 헤더를 명시적으로 관리합니다.
- 프록시 환경을 고려한 요청 처리 기준을 유지합니다.
- 인증 실패와 권한 부족 응답은 일관된 API 형태로 반환합니다.

### 정리

이 영역은 기능 추가보다 "일관성 유지"가 중요합니다. 프론트와 백엔드가 같은 인증/보안 규칙을 따라야 운영 중 예외가 줄어듭니다.

## 7. CSV / 텍스트 / 외부 입력 보호

### 문제

가계부 내보내기와 가져오기, 사용자 입력 텍스트, 외부 파일은 모두 외부 입력으로 간주해야 합니다. 이 입력은 수식 주입, 과도한 길이, 비정상 형식, 표시 오류 같은 문제를 만들 수 있습니다.

### 최종 유지 기준

- 내보내기 파일은 스프레드시트 실행 위험을 줄이는 방향으로 정리합니다.
- 텍스트 입력은 저장 전 정규화와 길이 제한을 거칩니다.
- 가져오기 데이터는 preview와 검증을 거쳐 반영합니다.
- 민감한 반출 데이터는 추가 보호 수단을 적용할 수 있게 유지합니다.

### 정리

이 영역의 기준은 "보기 편한 파일"보다 "안전하게 열 수 있는 파일", "무조건 넣는 import"보다 "검토 가능한 import"에 가깝습니다.

## 8. 업로드 / 스토리지 경로 보호

### 문제

파일 업로드는 경로 조작, 이상한 파일명, 스토리지 키 오염, 서버 메모리 집중 같은 위험을 동반합니다.

### 최종 유지 기준

- 업로드는 가능한 한 스토리지 직접 업로드와 서버 검증 중심 구조를 사용합니다.
- 파일명과 경로 조각은 안전한 형식으로 정규화합니다.
- 서버는 바이너리 전체를 오래 들고 있는 구조를 피합니다.

### 정리

업로드 보안의 핵심은 "어디까지를 브라우저가 하고, 어디부터 서버가 검증하는가"를 분리하는 데 있습니다.

## 9. 백업 / 복구 보호

### 문제

백업과 복구는 운영 도구 중 가장 파급력이 크기 때문에, 동시 실행, 잘못된 입력, 복구 중 일반 요청 유입 같은 문제가 모두 위험 요소가 됩니다.

### 최종 유지 기준

- 백업/복구는 단일 작업 기준으로 직렬화합니다.
- 복구 중에는 일반 서비스 요청을 제한하는 유지보수 게이트를 둡니다.
- 백업 아카이브 생성과 복원 입력은 경로 안전성을 검증합니다.
- 운영 실패 메시지는 필요한 수준만 외부에 드러내고 세부 내부 상태는 숨깁니다.

### 정리

이 영역은 "성공했는가"보다 "실패해도 더 큰 사고가 나지 않는가"가 더 중요합니다.

## 10. 운영 화면의 정보 노출 축소

### 문제

운영 편의를 위해 관리자 화면에 많은 정보를 실으면, 그만큼 민감한 상태 정보나 인증 관련 세부 정보가 한 화면에 드러날 수 있습니다.

### 최종 유지 기준

- 운영 목록은 페이지 단위로 나눠 필요한 범위만 확인합니다.
- 실패 사유와 상태 정보는 운영에 필요한 수준만 노출합니다.
- 공개 화면과 관리자 화면의 정보 밀도는 다르게 유지합니다.

### 정리

보안 패치는 방어 로직만이 아니라 "무엇을 보여주지 않을 것인가"까지 포함합니다.

## 11. 배포 / 인프라 보안 기준

### 현재 공개 가능한 기준

- 서비스는 HTTPS 기준으로 운영합니다.
- 외부 업로드나 스토리지 연동은 공개 가능한 안전한 주소 체계를 사용합니다.
- 데이터 계층과 애플리케이션 계층은 역할에 따라 분리 가능합니다.
- 권한 부여는 가능한 한 필요한 범위로 제한하는 원칙을 유지합니다.

### 정리

운영 인프라 보안은 한 번의 설정으로 끝나지 않고, 배포 문서와 실제 환경이 계속 일치하도록 관리하는 것이 중요합니다.

## 12. 현재 운영 원칙 요약

현재 공개 가능한 수준에서 정리한 운영 원칙은 아래와 같습니다.

- 계정 보호는 다단계 검증을 기본으로 한다.
- 반복 로그인 시도는 통제한다.
- 관리자 기능은 별도 보호 계층을 둔다.
- 요청 보호와 브라우저 보안 헤더를 유지한다.
- 외부 입력과 파일 반출은 sanitize / validate / protect를 기본 원칙으로 삼는다.
- 백업과 복구는 단일 실행과 유지보수 게이트를 기준으로 운영한다.

## 13. 검증 기준

보안 패치 성격의 변경은 가능한 한 아래 기준으로 검증합니다.

- 백엔드 테스트
- 프론트 빌드 검증
- 로그인 / 계정 변경 / 관리자 접근 수동 확인
- 내보내기 / 가져오기 / 업로드 흐름 확인
- 백업 / 복구와 유지보수 상태 전환 확인

구체적인 임계값, 내부 로그 기준, 탐지 규칙은 공개 문서에 남기지 않습니다.

## 14. 문서 원칙

이 문서는 공개 저장소와 위키에 올리는 문서이므로, 실제 운영 보호를 위해 아래 원칙을 유지합니다.

- 방어 방향은 기록하되 세부 수치는 공개하지 않는다.
- 내부 클래스명, 운영 절차, 임계값, 민감 경로는 필요 이상 적지 않는다.
- 공격자에게 도움이 되는 실패 조건 설명보다 운영 원칙을 우선한다.
