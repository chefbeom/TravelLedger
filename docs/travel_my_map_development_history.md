# Travel Map Development History

이 문서는 여행 지도, 사진 클러스터, 썸네일, fullscreen inspector, 큰 사진 보기에서 실제로 발생했던 문제와 최종 채택한 해결 방식을 정리한 문서입니다.

반복 시도나 중간 실험안은 줄이고, 마지막에 유지하기로 한 구현 기준을 중심으로 작성했습니다.

## 1. 초기 문제군

여행 지도와 사진 처리에서 실제로 크게 드러난 문제는 아래 다섯 축이었습니다.

1. 원본 이미지를 직접 다루는 과정에서 메모리 사용량이 급증하고 OOM이 발생함
2. 미리보기와 지도에서도 원본 사진이 fallback 되어 네트워크/메모리 부하가 커짐
3. prepared thumbnail 체계가 부족해서 용도별 이미지 응답이 일관되지 않음
4. 지도 클러스터/핀 UI가 재렌더 타이밍과 fullscreen 구조 때문에 반복적으로 깨짐
5. 클러스터 상세, 사진 목록, 큰 사진 보기 범위가 뒤섞여 UX와 성능이 모두 불안정함

## 2. OOM과 원본 응답 문제 해결

### 증상

- `travel/media/.../content` 요청에서 Java heap space OOM 발생
- 예외 처리 중 `image/jpeg` content-type 상태에서 JSON 예외 응답이 섞이며 2차 오류 발생

### 원인

- 원본 이미지를 `readAllBytes()`로 한 번에 메모리에 올리던 경로 존재
- prepared thumbnail이 없을 때 요청 시점에 원본을 다시 읽어 즉석 썸네일 생성
- 실패 상황에서 이미지 응답과 JSON 예외 응답이 충돌

### 최종 해결 방식

- 원본 응답은 스트리밍 기반으로 전환
- prepared thumbnail이 없더라도 일반 미리보기에서 원본을 바로 내리지 않도록 정책 변경
- 예외 응답은 항상 JSON content-type으로 강제

### 결과

- OOM 재현 경로 제거
- 이미지 응답 실패 시 예외 처리 안정화
- 원본은 크게 보기/다운로드에서만 사용한다는 원칙이 명확해짐

## 3. 원본 fallback 제거와 페이징

### 증상

- 선택된 클러스터, 선택한 사진 정보, 클러스터 포함 사진, 가족 사진첩 등에서 원본 이미지가 fallback 되어 로딩이 무거움
- 사진첩과 클러스터 목록이 한 번에 많이 내려와 초기 로드 부하가 큼

### 최종 해결 방식

- 미리보기/목록/클러스터 영역은 prepared thumbnail만 사용
- 원본은 크게 보기와 다운로드에서만 사용
- 가족 앨범과 클러스터 사진 목록은 paging / lazy loading 적용

### 결과

- 일반 화면에서 원본 이미지 직접 응답 차단
- 초기 로딩과 스크롤 구간 부하 감소

## 4. Prepared Thumbnail 체계 정리

### 문제

용도별로 필요한 이미지 크기가 달랐고, 그때그때 리사이징하면 응답 시간이 흔들렸습니다.

### 최종 프로필

- `pin`
- `mini`
- `preview`
- `detail`

### 최종 해결 방식

- 용도별 prepared thumbnail 프로필을 고정
- MinIO에 prepared thumbnail을 저장하고 프론트는 필요한 프로필만 요청
- 예전 사진은 백필 또는 안전망으로 보정

### 결과

- 지도 핀, 팝업, inspector, 카드, detail 미리보기에서 일관된 크기 사용
- 원본/썸네일 역할 분리

## 5. 썸네일 생성 책임 이동

### 문제

백엔드가 원본 이미지 바이너리와 썸네일 생성까지 직접 처리하면 부하 집중점이 됩니다.

### 최종 해결 방식

여행 업로드 기준으로 구조를 아래처럼 이동했습니다.

1. 프론트가 원본에서 필요한 썸네일을 준비
2. 프론트가 presigned URL 요청
3. 브라우저가 MinIO로 직접 업로드
4. 백엔드는 업로드 검증과 메타데이터 저장만 수행

### 결과

- 백엔드가 원본 이미지 업로드 본문을 직접 받지 않는 경로 정착
- 이미지 바이너리 처리 부하가 스토리지 직접 업로드로 분산됨

## 6. 썸네일 백필 배치

### 문제

예전 사진은 prepared thumbnail이 없는 상태라 지도 첫 진입 시점에 보정 비용이 몰릴 수 있었습니다.

### 최종 해결 방식

- 앱 시작 후 1회 + 주기 스케줄로 백필 배치 수행
- 페이지 단위로 사진을 읽고, 누락된 prepared thumbnail만 생성
- 신규 업로드는 업로드 시점에 준비, 예전 데이터는 배치가 따라잡는 방식 사용

### 결과

- 지도 진입 전에 대부분의 예전 데이터가 준비됨
- 요청 시점 생성 비중 축소

## 7. 과도한 쿼리와 `/api/api` 문제

### 증상

- 동일 미디어 메타데이터 쿼리가 반복
- 클러스터 상세 요청이 `/api/api/...`로 잘못 나가며 404 발생

### 원인

- 보호된 이미지 엔드포인트에서 동일한 접근 검증 쿼리가 반복
- paged API URL 조합 시 `/api`가 중복 결합됨

### 최종 해결 방식

- 미디어 메타데이터 캐시 적용
- 잘못된 API path 조합 수정

### 결과

- 운영 로그 노이즈 감소
- 클러스터 상세 404 제거

## 8. 프론트 메모리 최적화

### 증상

- 여행 지도/요약 화면에서 프론트 메모리 사용량이 비정상적으로 커짐

### 원인

- 같은 사진 데이터를 여러 컴포넌트가 중복 보관
- 지도 렌더에 필요 없는 `mediaItems` 전체 배열 전달
- deep watch와 재정렬/복사 비용 누적

### 최종 해결 방식

- 지도에 필요한 최소 데이터만 전달
- 대표 썸네일 URL과 개수 중심으로 축소
- 중복 정렬/복사 제거
- deep watch 축소

### 결과

- 메모리 사용량과 렌더 비용 감소

## 9. 클러스터/핀 렌더와 선택 흐름

### 문제

- 확대/축소 시 클러스터가 계속 다시 묶이고 풀리며 부하가 큼
- 핀 보기에서 텍스트 기반 핀이 직관적이지 않음

### 최종 해결 방식

- 서버가 계산한 클러스터를 고정적으로 사용
- 확대/축소 때 프론트가 다시 군집 계산하지 않음
- 핀 보기는 작은 썸네일 핀으로 변경

### 결과

- 줌 시 재계산 부하 감소
- 핀 보기가 사진 중심으로 직관화

## 10. 클러스터 상세와 선택 상태 복구

### 증상

- 지도는 보이는데 클러스터 상세 404 하나 때문에 패널 전체가 깨짐
- 클러스터 상세와 summary snapshot이 잠깐 어긋나면 선택 정보가 비어버림

### 최종 해결 방식

- 개요 로드 실패와 상세 로드 실패를 분리
- 클러스터 상세가 없으면 snapshot 재생성 후 다시 탐색
- 선택된 클러스터는 있는데 상세가 비어 있으면 자동 복구 로드

### 결과

- 지도 자체는 유지
- 상세만 복구 재시도

## 11. popup 두 번 클릭 문제

### 반복 증상

- 클러스터된 핀/이미지를 한 번 누르면 popup 또는 정보창이 닫히고, 같은 대상을 두 번 눌러야 정상 표시됨

### 원인

- marker click과 선택 상태 변경, detail 재로드, marker rerender가 같은 프레임에서 겹침
- 첫 클릭에 열린 popup이 이전 marker 인스턴스에 붙은 채 사라짐
- 선택된 photo id 변화가 현재 보기 모드와 무관하게 marker layer 재생성을 유발

### 마지막에 채택한 해결 방식

- popup 재오픈은 render 후 실제 marker 인스턴스를 다시 찾아 재시도하는 방식으로 유지
- 동시에 선택 watch는 현재 display mode에서 필요한 선택 키만 보도록 축소
- cluster 모드에서 detail 로드 때문에 `selectedPhotoId`가 바뀌어도 marker layer 전체를 다시 갈아엎지 않게 조정

### 결과

- 첫 클릭에서 닫히는 race condition이 크게 줄어듦
- popup과 오른쪽 inspector의 선택 흐름이 더 안정적으로 맞춰짐

## 12. fullscreen에서 lightbox가 안 보이던 문제

### 증상

- 일반 화면에서는 큰 사진이 보이는데 fullscreen에서는 모달이 열려도 화면에 안 보임

### 원인

- fullscreen 시 브라우저는 fullscreen element subtree만 렌더
- lightbox가 fullscreen subtree 밖에서 렌더되면 DOM에 있어도 화면에는 안 보임

### 최종 해결 방식

- fullscreen일 때 lightbox를 지도 fullscreen subtree 안에서 렌더

### 결과

- fullscreen에서도 큰 사진 정상 표시

## 13. fullscreen 오른쪽 inspector overflow

### 반복 증상

- 오른쪽 패널이 아래로 범위를 뚫고 내려감
- 클러스터 사진이 많을수록 panel 바깥까지 밀려남

### 최종 원인

- inspector는 `travel-map__overlay` 안에 렌더되는데, 높이는 `100dvh` 기준으로 계산하고 있었음
- 실제 사용 가능 높이는 toolbar와 hint를 제외한 overlay 높이인데, 패널이 viewport 전체 높이를 기준으로 더 크게 잡혀 아래로 넘침

### 마지막 해결 방식

- overlay 자체를 `overflow: hidden`으로 고정
- overlay 자식에 `max-height: 100%` 적용
- fullscreen inspector 높이를 `100dvh`가 아니라 overlay 기준 `height: 100%`, `max-height: 100%`로 변경
- 실제 스크롤은 panel 내부 gallery/grid 영역만 담당

### 결과

- inspector가 지도 fullscreen 범위를 넘지 않음
- 내부에서만 스크롤

## 14. 클러스터 내부 사진 UI 정리

### 요구 변화

- 작은 카드에서 파일명/날짜 텍스트 제거
- 이미지만 보이게 하고, 이미지를 누르면 크게 보기
- 대표 사진 지정은 작은 카드가 아니라 크게 열린 사진에서만 수행

### 최종 해결 방식

- 클러스터 내부 카드에서 텍스트와 액션 버튼 제거
- 이미지 블록만 유지
- 이미지 클릭 시 cluster-scope lightbox open
- 대표 사진 지정은 lightbox 내부 액션 버튼으로 이동

### 결과

- inspector 밀도 향상
- 조작 지점 단순화

## 15. 큰 사진 이전/다음 이동 범위 정리

### 요구 변화

- 지도에서 직접 연 큰 사진은 전체 지도 사진 기준으로 이동
- 클러스터 내부 블록에서 연 큰 사진은 그 클러스터 내부 사진끼리만 이동

### 최종 해결 방식

- lightbox source를 `global` / `cluster` scope로 분리
- `global`: 지도 overview의 `photoPins` 기반 전체 사진 목록
- `cluster`: 선택된 클러스터 내부 사진 목록
- 두 목록 모두 시간순 정렬 후 lightbox에 주입
- cluster scope일 때만 대표 사진 지정 버튼 노출

### 결과

- 큰 사진 이동 범위가 사용자의 진입 위치와 일치
- 대표 사진 지정 컨텍스트도 클러스터 내부로 한정

## 16. 검증 방식

매 단계마다 가능한 한 아래 기준으로 검증했습니다.

- 프론트: `cmd /c npm run build`
- 백엔드: `.\gradlew.bat test`
- 지도 수동 검증
  - 첫 클릭에 popup/정보창 갱신
  - fullscreen에서 큰 사진 노출
  - inspector 범위 초과 없음
  - 클러스터 내부 사진 paging / lazy loading 동작
  - 대표 사진 지정 후 반영 확인

## 17. 현재 운영 기준 요약

현재 여행 지도/사진 관련 운영 기준은 아래와 같습니다.

- 여행 업로드는 프론트 준비 + 스토리지 직접 업로드 + 백엔드 메타데이터 저장
- 일반 미리보기는 prepared thumbnail 사용, 원본은 크게 보기/다운로드에 한정
- 예전 사진은 백필 배치로 prepared thumbnail 보정
- 지도는 서버 계산 클러스터를 고정적으로 사용
- fullscreen inspector는 지도 overlay 범위 안에서만 동작
- 큰 사진 이동은 `global` 과 `cluster` scope로 분리
